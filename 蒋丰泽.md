# 2.第二阶段：docker的实现

Docker 是一个开源的容器化平台，它将应用程序及其依赖打包进轻量级、可移植的镜像中，通过容器提供隔离的运行环境。

**核心用途：**

- **环境标准化**：将应用与依赖整体打包，确保开发、测试、生产环境完全一致，消除“环境差异”问题。
- **资源高效隔离**：共享宿主机内核，但为每个容器提供独立的运行空间，多应用互不干扰，资源利用率远超虚拟机。
- **快速部署交付**：镜像可秒级启动，支持快速分发、弹性扩缩、持续集成与交付，适用于微服务和云原生架构。

### 二、7 种 Namespace 类型

所有 7 种 Linux 标准 namespace 均已实现，每种有独立的 `ROOT_*` 静态根实例。

#### 1. UTS Namespace — `uts.rs`
```rust
pub struct UtNamespace {
    pub nodename: [c_char; 65],     // hostname，默认 "starry"
    pub domainname: [c_char; 65],   // domain name
}
```
**实现**：`clone_ns()` 直接拷贝字节数组，隔离 hostname/domainname。

#### 2. IPC Namespace — `ipc.rs`
```rust
pub struct IpcNamespace {
    pub ns_id: u64,  // 由 NEXT_IPC_NS_ID 原子计数器分配
}
```
**实现**：通过 `ns_id` 实现 System V IPC 隔离。共享内存和消息队列的资源查找键为 `(key, ns_id)` 二元组。

#### 3. Mount Namespace — `mnt.rs`
```rust
pub struct MntNamespace {
    pub ns_id: u64,
    pub root: [c_char; 256],  // 默认 "/"
}
```
**特殊之处**：
- 挂载树本身由 `ax-fs-ng` 的 `FsContext` 持有，且是**进程作用域**
- `unshare(CLONE_NEWNS)` → 复制挂载树
- `setns(CLONE_NEWNS)` → 使用 NsFd 中携带的目标 FsContext
- `clone(CLONE_NEWNS)` → 子进程获得已解耦的 FsContext
- `pivot_root(2)` → 仅作用于当前进程的 FS_CONTEXT

#### 4. PID Namespace — `pid.rs`
```rust
pub struct PidNamespace {
    pub level: u32,                        // 嵌套层级（根 = 0）
    next_pid: u32,                         // 下一个本地 PID
    pid_map: BTreeMap<u64, u32>,           // 全局 TID → 本地 PID
}
```

| 方法                          | 功能                                          |
| ----------------------------- | --------------------------------------------- |
| `alloc_local_pid(global_tid)` | 分配命名空间本地 PID，注册到 pid_map          |
| `local_pid(global_tid)`       | 查表解析本地 PID（根层级直接返回 global_tid） |

**关键设计**：`clone(CLONE_NEWPID)` 时，子进程不会立即切换 PID namespace，而是放入 `NsProxy.child_pid_ns` 字段。子进程启动后消费这个挂起的 namespace，成为新 namespace 中的 PID 1。

#### 5. Network Namespace — `net.rs`
```rust
pub struct NetNamespace {
    pub ns_id: u64,  // 根 ns_id = 0
}
```
通过 `in_root_net_ns()` 判断：
- 根 net ns（ns_id == 0）：可见 eth0 等真实设备
- 非根 net ns：仅可见 lo 回环设备

#### 6. User Namespace — `user.rs`
```rust
pub struct UserNamespace {
    pub owner_uid: u32,
    pub is_root: bool,
    pub uid_mapped: bool,   // 是否已写入 /proc/<pid>/uid_map
    pub gid_mapped: bool,   // 是否已写入 /proc/<pid>/gid_map
}
```
对未完成映射的一方，`sys_getuid/geteuid/getgid/getegid` 返回 **65534**（nobody / overflow UID）。

#### 7. Cgroup Namespace
**模拟**：`NsFd::Cgroup` 无数据，`CLONE_NEWCGROUP` 被静默接受，但 cgroup v2 层级实际是**全局的**。

### 三、NsProxy 聚合器 — `lib.rs`

```rust
pub struct NsProxy {
    pub uts_ns:  Arc<SpinNoIrq<UtNamespace>>,
    pub ipc_ns:  Arc<SpinNoIrq<IpcNamespace>>,
    pub mnt_ns:  Arc<SpinNoIrq<MntNamespace>>,
    pub pid_ns:  Arc<SpinNoIrq<PidNamespace>>,
    pub child_pid_ns: Option<Arc<SpinNoIrq<PidNamespace>>>,
    pub net_ns:  Arc<SpinNoIrq<NetNamespace>>,
    pub user_ns: Arc<SpinNoIrq<UserNamespace>>,
}
```

**核心方法**：

| 方法                     | 用途                                     | 调用者                  |
| ------------------------ | ---------------------------------------- | ----------------------- |
| `new_root()`             | 创建指向所有 7 种根 namespace 的 NsProxy | 进程初始化              |
| `clone_all()`            | 浅拷贝所有 Arc 引用                      | fork()（无 CLONE_NEW*） |
| `unshare_*()`            | 替换对应 Arc 为克隆体                    | unshare(2)              |
| `prepare_child_pid_ns()` | 放入 child_pid_ns                        | unshare(CLONE_NEWPID)   |
| `set_ns_*()`             | 替换为已有的 namespace 实例              | setns(2)                |

**设计模式**：`Arc<SpinNoIrq<T>>` + **写时复制** — `clone_all()` 增加引用计数，`unshare_*()` 创建新实例。

### 四、系统调用实现

#### 1. unshare(2) — `namespace.rs`
- 根据标志位调用对应的 `unshare_*()` 方法
- 支持：`CLONE_NEWUTS | NEWPID | NEWNS | NEWNET | NEWIPC | NEWUSER | NEWCGROUP | CLONE_FS`
- `CLONE_NEWNS`：多线程时检查完整性 + 调用 `FsContext::unshare_mount_namespace()`
- `CLONE_NEWPID`：调用 `prepare_child_pid_ns()`，**不立即生效**（延迟到子进程）

#### 2. setns(2) — `namespace.rs`

| 路径           | fd 来源                         | 说明                                                 |
| -------------- | ------------------------------- | ---------------------------------------------------- |
| **NsFd 路径**  | `open("/proc/<pid>/ns/<type>")` | nstype 必须为 0 或匹配 fd 类型；PID/USER ns 需单线程 |
| **PidFd 路径** | `pidfd_open(2)`                 | nstype 是位掩码，一次可加入多个 namespace            |

#### 3. clone(2) — `clone.rs`
1. 从父进程复制 nsproxy
2. 对每个 `CLONE_NEW*` 标志调用对应的 `unshare_*()`
3. `CLONE_NEWPID` 额外调用 `alloc_local_pid()`
4. 消费父进程的 `child_pid_ns`（来自 `unshare(CLONE_NEWPID)`）

### 五、与进程模型的集成

```rust
pub struct ProcessData {

    // ...
```

- 创建时调用 `NsProxy::new_root()` 初始化

### 六、/proc 与 NsFd 机制

`/proc/<pid>/ns/` 目录暴露 7 个条目：

```
/proc/<pid>/ns/
  ├── uts     → "uts:[<hostname>]"
  ├── ipc     → "ipc:[<ns_id>]"
  ├── mnt     → "mnt:[<ns_id>]"
  ├── pid     → "pid:[<level>]"      ← 显示层级而非 id
  ├── net     → "net:[<ns_id>]"
  ├── user    → "user:[<owner_uid>]"
  └── cgroup  → "cgroup:[root]"
```

**NsFd 枚举**：
```rust
pub enum NsFd {
    Uts(Arc<SpinNoIrq<UtNamespace>>),
    Ipc(Arc<SpinNoIrq<IpcNamespace>>),
    Mnt { namespace: Arc<SpinNoIrq<MntNamespace>>, fs_context: FsContext },
    Pid(Arc<SpinNoIrq<PidNamespace>>),
    Net(Arc<SpinNoIrq<NetNamespace>>),
    User(Arc<SpinNoIrq<UserNamespace>>),
    Cgroup,
}
```

`Mnt` 变体特殊：额外携带 `FsContext`，在 `setns(CLONE_NEWNS)` 时切换整个挂载树。

| 特性                | 实现状态                                      |
| ------------------- | --------------------------------------------- |
| UTS namespace       | ✅ 完整 — hostname/domainname 隔离             |
| IPC namespace       | ✅ 完整 — 通过 ns_id 过滤 SysV IPC 资源        |
| Mount namespace     | ✅ 完整 — 进程作用域 FsContext，支持传播标志   |
| PID namespace       | ✅ 完整 — 嵌套层级 + BTreeMap 本地 PID 映射    |
| Network namespace   | ✅ 基础 — 通过 in_root_net_ns() 控制设备可见性 |
| User namespace      | ✅ 完整 — UID/GID 映射跟踪 + overflow UID 返回 |
| Cgroup namespace    | ⚠️ 模拟 — 接受但不隔离 cgroup v2 层级          |
| clone/unshare/setns | ✅ — 三种系统调用全部支持                      |

## 2.3seccomp的实现

## StarryOS Seccomp 实现详解

### 一、总体概览

**实现位置**：`kernel/src/task/seccomp.rs`

**核心组成**：

| 组件                        | 说明                                            |
| --------------------------- | ----------------------------------------------- |
| **经典 BPF 解释器（cBPF）** | 纯软件实现，支持完整 BPF 指令集                 |
| **三种模式**                | Disabled / Strict / Filter                      |
| **7 种 SECCOMP_RET 动作**   | 按 Linux 优先级规则求值                         |
| **每线程状态**              | 通过 clone/fork 继承                            |
| **两种入口**                | `seccomp(2)` 系统调用 + `prctl(PR_SET_SECCOMP)` |

---

### 二、核心数据结构

#### SeccompState — 每线程状态

```rust
pub struct SeccompState {
    mode: SeccompMode,
    filters: Vec<SeccompFilter>,  // 从旧到新追加
}

enum SeccompMode {
    Disabled,   // 默认：所有系统调用放行
    Strict,     // 仅允许 read/write/exit/exit_group/rt_sigreturn
    Filter,     // 由 BPF 过滤器程序决定
}
```

`Thread` 结构体中持有：`seccomp: SpinNoIrq<SeccompState>`

#### BPF 指令结构（与 Linux ABI 兼容）

```rust
#[repr(C)]
pub struct SockFilter {
    pub code: u16,   // BPF 操作码
    pub jt: u8,      // 条件为真时的跳转偏移
    pub jf: u8,      // 条件为假时的跳转偏移
    pub k: u32,      // 常量操作数
}

#[repr(C)]
pub struct SockFprog {
    pub len: u16,
    pub filter: *const SockFilter,
}
```

#### SeccompData — BPF 可见的数据记录

```rust
struct SeccompData {
    nr: i32,                    // 系统调用号
    arch: u32,                  // AUDIT_ARCH_* 架构标识
    instruction_pointer: u64,   // 执行系统调用的 PC
    args: [u64; 6],             // 6 个系统调用参数
}
```

| 架构        | AUDIT_ARCH  |
| ----------- | ----------- |
| x86_64      | 0xC000_003E |
| aarch64     | 0xC000_00B7 |
| riscv64     | 0xC000_00F3 |
| loongarch64 | 0xC000_0102 |

### 三、系统调用调度路径

**关键集成点**：`syscall/mod.rs` 的 `handle_syscall()` 中，每次系统调用分发前执行 seccomp 评估：

```rust
pub fn handle_syscall(uctx: &mut UserContext) {
    // ★ 在这里评估 seccomp
    match ax_task::current().as_thread().seccomp_state().evaluate(uctx) {
        SeccompDecision::Allow => { /* 继续正常分发 */ }
        SeccompDecision::Errno(e) => { uctx.set_retval(seccomp_errno(e)); return; }
        SeccompDecision::KillProcess => { do_exit(SIGSYS, true); return; }
        SeccompDecision::KillThread => { do_exit(SIGSYS, false); return; }
        SeccompDecision::UnsupportedAction => { uctx.set_retval(-ENOSYS); return; }
    }
    // ... 分发到真正的系统调用处理器 ...
}
```

**设计细节**：`seccomp_state()` 返回 `SeccompState` 的克隆，锁持有时间仅限 clone 操作，BPF 执行期间不持锁。

---

### 四、经典 BPF 解释器

#### 虚拟寄存器

| 寄存器    | 说明                |
| --------- | ------------------- |
| `a`       | 累加器（32 位）     |
| `x`       | 索引寄存器（32 位） |
| `mem[16]` | 16 槽位暂存内存     |

#### 实现的指令类

| 指令类               | 支持的操作                                                   |
| -------------------- | ------------------------------------------------------------ |
| `BPF_LD`             | 立即数加载、seccomp_data 绝对偏移加载（W/H/B）、暂存内存加载、包长度加载 |
| `BPF_LDX`            | 同上（目标为 x 寄存器）                                      |
| `BPF_ST` / `BPF_STX` | 存储 a / x 到暂存内存槽位                                    |
| `BPF_ALU`            | ADD, SUB, MUL, DIV, OR, AND, LSH, RSH, NEG, MOD, XOR         |
| `BPF_JMP`            | JA（无条件跳转）、JEQ, JGT, JGE, JSET（条件跳转）            |
| `BPF_RET`            | 返回 a 或立即数 k                                            |
| `BPF_MISC`           | TAX（a → x）、TXA（x → a）                                   |

**安全姿态**：**fail-closed** — 遇到不支持的操作码、越界内存访问、非法跳转目标、除零/模零等错误时，统一返回 `SECCOMP_RET_KILL_THREAD`。

**约束**：
- 最大 4096 条指令（`BPF_MAXINSNS = 4096`）
- 16 个暂存内存槽位（`BPF_MEMWORDS = 16`）

#### BPF_ABS 加载映射

| 偏移                   | 字段                         |
| ---------------------- | ---------------------------- |
| 0                      | 系统调用号（nr 低 32 位）    |
| 4                      | 架构（arch）                 |
| 8                      | instruction_pointer 低 32 位 |
| 12                     | instruction_pointer 高 32 位 |
| 16, 24, 32, 40, 48, 56 | args[0..5] 低 32 位          |
| 20, 28, 36, 44, 52, 60 | args[0..5] 高 32 位          |

支持 `BPF_W`（32 位）、`BPF_H`（16 位）、`BPF_B`（8 位）加载。

### 五、SECCOMP_RET 动作与优先级

| 常量                                    | 优先级   | 实际决策                 | 说明                       |
| --------------------------------------- | -------- | ------------------------ | -------------------------- |
| `SECCOMP_RET_KILL_PROCESS` (0x80000000) | 最高 (7) | KillProcess              | 整个进程退出               |
| `SECCOMP_RET_KILL_THREAD` (0x00000000)  | 6        | KillThread               | 仅当前线程退出             |
| `SECCOMP_RET_TRAP` (0x00030000)         | 5        | UnsupportedAction→ENOSYS | ⚠️ 未模拟（应发 SIGSYS）    |
| `SECCOMP_RET_ERRNO` (0x00050000)        | 4        | Errno(data)              | 返回负错误码               |
| `SECCOMP_RET_TRACE` (0x7ff00000)        | 3        | UnsupportedAction→ENOSYS | ⚠️ 未模拟（应通知 ptracer） |
| `SECCOMP_RET_LOG` (0x7ffc0000)          | 2        | Allow                    | 视为 ALLOW（无审计日志）   |
| `SECCOMP_RET_ALLOW` (0x7fff0000)        | 最低 (1) | Allow                    | 正常放行                   |

**多过滤器求值**：从最新到最旧依次执行，按优先级合并结果（与 Linux 一致：最高优先级的动作胜出）。

---

### 六、系统调用接口

#### seccomp(2) — `syscall/sys.rs`

| 操作                           | 功能                     |
| ------------------------------ | ------------------------ |
| `SECCOMP_SET_MODE_STRICT` (0)  | 启用严格模式             |
| `SECCOMP_SET_MODE_FILTER` (1)  | 安装 BPF 过滤器          |
| `SECCOMP_GET_ACTION_AVAIL` (2) | 查询某个返回动作是否可用 |

**安装流程（FILTER 模式）**：
1. `check_seccomp_install_permission()` — 必须满足 `no_new_privs` 或 `CAP_SYS_ADMIN`
2. `read_seccomp_filter(args)` — 从用户空间读取 SockFprog，验证，拷贝指令
3. `thread.append_seccomp_filter(filter)` — 追加到过滤器列表，切换到 Filter 模式
4. 若 `SECCOMP_FILTER_FLAG_TSYNC` 置位 → `sync_seccomp_to_thread_group()`

#### prctl(PR_SET_SECCOMP) — `syscall/task/ctl.rs`

直接路由到 `sys_seccomp()`，flags=0（prctl 不支持 TSYNC）。

### 七、继承与同步

**Clone/Fork 继承**（`clone.rs`）：
```rust
thr.set_seccomp_state(curr_thread.seccomp_state());
```
每个新线程完整拷贝父线程的 seccomp 状态（所有过滤器和模式）。

**TSYNC 同步**：安装过滤器时若带有 `SECCOMP_FILTER_FLAG_TSYNC`，遍历当前进程的所有兄弟线程并写入相同的 SeccompState。

| 特性                     | 状态         | 影响                             |

| `SECCOMP_IOCTL_NOTIF_*`  | ❌ 未实现     | ioctl 通知接口                   |
| cBPF → eBPF 转换         | ❌ 未实现     | 仅支持经典 BPF                   |

### 十、架构总览

```
用户空间
  │
  ├─ seccomp(SECCOMP_SET_MODE_FILTER, ...)
  ├─ prctl(PR_SET_SECCOMP, ...)
  └─ 系统调用
       │
       ▼
syscall/mod.rs: handle_syscall()
  │
  ├─ Thread::seccomp_state()        ← 克隆 SeccompState（不持锁）
  │
  ├─ SeccompState::evaluate(uctx)
  │   ├─ Strict  → 检查 read/write/exit/exit_group/rt_sigreturn
  │   └─ Filter  → 遍历 filters[]，逐个执行 BPF 解释器
  │       └─ SeccompFilter::execute(seccomp_data)
  │           ├─ 载入 uctx → SeccompData
  │           └─ cBPF 解释器循环（4096 指令上限）
  │
  ├─ SeccompDecision::Allow  → 继续分发
  ├─ SeccompDecision::Errno  → 返回 -errno
  ├─ SeccompDecision::KillProcess → do_exit(SIGSYS, group=true)
  ├─ SeccompDecision::KillThread  → do_exit(SIGSYS, group=false)
  └─ SeccompDecision::Unsupported → -ENOSYS
```

**核心设计原则**：
- **fail-closed**：错误即拒绝
- **每线程状态**：符合 Linux
- **纯软件 cBPF 解释器**：安全优先

## 2.4overlayfs实现

## StarryOS Overlay 实现详解

- **OverlayFS 文件系统**（StarryOS 内核）— Linux 兼容的联合挂载文件系统

## 一、OverlayFS 文件系统（StarryOS 内核）

### 1. 架构模型

```
用户视角（merged 目录）
  ├── file_a.txt      ← 来自 upper
  ├── file_b.txt      ← 来自 lower（被 copy-up 到 upper 后修改）
  └── dir1/
        ├── file_c    ← upper 覆盖 lower 同名文件
        └── opaque/   ← upper 中的 opaque 目录，隐藏 lower 所有子项

实际存储：
  upper/  (可写层)
  lower/  (只读层)
  work/   (内部工作目录，Linux ABI 要求)
```

### 2. 核心数据结构

```rust
// 挂载选项
pub struct OverlayOptions {
    pub lower_dirs: Vec<Location>,     // lower 层（从上到下排列）
    pub upper_dir: Option<Location>,   // 可写 upper 层（None = 只读模式）
    pub work_dir: Option<Location>,    // 工作目录（有 upper 时必须）
}

// VFS 注册
OverlayFs  // FilesystemOps trait，魔数 0x794c7630，名称 "overlay"
```

### 3. 目录合并逻辑（OverlayDir）

**readdir 合并算法**（从底到顶逐层合并）：
1. 从最底层的 lower 开始收集条目
2. 每一层 upper（包括 upper 目录本身）覆盖同名条目
3. 空白（whiteout）条目 → 删除 lower 中的对应名称
4. opaque 目录（包含 `.wh..wh..opq` 标记）→ 完全跳过该层 lower 的所有子项

**查找（lookup）**：先查 upper，若 upper 有 whiteout（字符设备 0:0）则隐藏 lower；若 upper 目录有 opaque 标记则阻止 lower 可见。

**创建**：验证无可见同名条目，移除已有 whiteout，在 upper 中创建。

**删除（unlink）**：若 lower 中存在对应条目，则在 upper 中创建 whiteout；若仅在 upper 中存在，则直接删除。

**重命名**：lower 文件执行 copy-up 后再重命名；拒绝 lower 目录的重命名（返回 EXDEV）。

### 4. 惰性 Copy-up 引擎

**触发条件**：
- write() → copy_up
- append() → copy_up
- set_len() → copy_up
- set_symlink() → copy_up
- ioctl() → copy_up
- read() → 不触发 copy-up

**copy_entry() 步骤**：

1. 在 upper 创建对应类型条目（文件/目录/符号链接）
2. `copy_metadata()` — 保留 mode、owner、rdev、atime、mtime
3. `copy_file_contents()` — 4096 字节块逐块复制

### 5. Whiteout 与 Opaque 机制

| 机制     | 实现                               | 用途                   |
| -------- | ---------------------------------- | ---------------------- |
| Whiteout | 字符设备 makedev(0,0)              | 标记 lower 文件被删除  |
| Opaque   | upper 目录中的 `.wh..wh..opq` 文件 | 标记整个目录屏蔽 lower |

### 6. xattr 集成

| 操作      | Overlay 处理                                                 |
| --------- | ------------------------------------------------------------ |
| 读取/列出 | `visible_target()` → 解析到当前可见的实际位置                |
| 设置/删除 | `ensure_copy_up_target()` → 先将 lower 文件 copy-up 到 upper，再写入 xattr |

### 7. Mount 系统调用集成

```rust
"overlay" => {
    let (lower_paths, upper_path, work_path) = parse_overlay_options(data)?;
    // 支持的选项：
    //   lowerdir=<colon-separated paths>   (必需)
    //   upperdir=<path>                    (可选；有则必须 workdir)
    //   workdir=<path>                     (可选；有则必须 upperdir)
    //   index=off                          (仅接受 "off")
    //   redirect_dir=off                   (仅接受 "off")
}
```

**两种模式**：
- 只读模式：仅 lowerdir，无 upperdir/workdir
- 读写模式：lowerdir + upperdir + workdir

### 8. 节点缓存策略

Overlay 条目统一设置 `NodeFlags::NON_CACHEABLE`，确保每次查找都重新评估合并视图。

## 2.5capabilities 的实现

## StarryOS Capabilities 实现详解

### 一、总体架构

Capabilities 系统将传统 root 的超级权限拆分为独立的、可分配的能力单元。StarryOS 实现了完整的 **Linux POSIX capabilities V3 ABI**。

```
用户空间
  │
  ├── capget(2) / capset(2)     ← 读取/设置能力集合
  ├── prctl(PR_CAP_*, ...)      ← 环境能力操作、边界集控制
  └── /proc/<pid>/status        ← 查看五个能力集的十六进制
       │
       ▼
系统调用层
  ├── sys_capget() / sys_capset()     [syscall/task/ctl.rs]
  └── sys_prctl()                     [syscall/task/ctl.rs]
       │
       ▼
凭证层
  └── Cred 结构体                      [task/cred.rs]
       └── 5 个 u64 位图 + keep_caps 标志
       │
       ▼
能力检查点 (~20 处)
  ├── 文件系统: CAP_CHOWN, CAP_FOWNER
  ├── 网络:     CAP_NET_RAW
  ├── UID/GID:  CAP_SETUID, CAP_SETGID
  ├── 调度:     CAP_SYS_NICE
  ├── 资源:     CAP_SYS_RESOURCE
  ├── 系统:     CAP_SYS_BOOT, CAP_SYS_ADMIN
  └── 信号:     CAP_KILL (euid==0 近似)
```

---

### 二、核心数据结构 — `task/cred.rs`

#### Cred 结构体

```rust
pub struct Cred {
    // POSIX UID/GID
    pub uid: u32, pub gid: u32,       // 真实
    pub euid: u32, pub egid: u32,     // 有效
    pub suid: u32, pub sgid: u32,     // 保存的 set-user/set-group
    pub fsuid: u32, pub fsgid: u32,   // 文件系统
    pub groups: Arc<[u32]>,           // 附加组

    // 五个 capabilities 集合（u64 位图）
    pub cap_inheritable: u64,   // 可继承集 — 通过 exec 传递
    pub cap_permitted: u64,     // 允许集 — 能力的上限
    pub cap_effective: u64,     // 有效集 — 当前实际有效的能力
    pub cap_bounding: u64,      // 边界集 — 所有集合的绝对上限 (cap_bset)
    pub cap_ambient: u64,       // 环境集 — 非特权 exec 中保留的能力

    // 标志
    pub keep_caps: bool,        // PR_SET_KEEPCAPS — UID 切换时保留允许集
}
```

#### CAP_* 常量（部分）

| 常量             | 编号 | 用途                   |
| ---------------- | ---- | ---------------------- |
| CAP_CHOWN        | 0    | 更改文件所有者         |
| CAP_DAC_OVERRIDE | 1    | 绕过读/写/执行权限检查 |
| CAP_FOWNER       | 3    | 绕过文件所有者检查     |
| CAP_SETUID       | 7    | 任意 setuid 操作       |
| CAP_SETGID       | 6    | 任意 setgid 操作       |
| CAP_SETPCAP      | 8    | 修改其他进程的能力集   |
| CAP_SYS_ADMIN    | 21   | 通用系统管理           |
| CAP_SYS_BOOT     | 23   | 重启系统               |
| CAP_SYS_NICE     | 22   | 修改调度优先级         |
| CAP_SYS_RESOURCE | 24   | 提升资源限制           |
| CAP_NET_RAW      | 13   | 原始套接字             |
| CAP_SYS_PTRACE   | 19   | 跟踪任意进程           |

#### 关键方法

| 方法                           | 说明                                             |
| ------------------------------ | ------------------------------------------------ |
| `Cred::root()`                 | uid/gid=0, permitted=effective=bounding=CAP_MASK |
| `Cred::unprivileged(uid, gid)` | 普通配置，bounding 保持 CAP_MASK                 |
| `has_cap(&self, cap)`          | 检查 effective 集中是否有某能力                  |
| `has_cap_setuid()/setgid()`    | 检查对应能力                                     |
| `has_cap_net_raw()`            | 检查 CAP_NET_RAW                                 |
| `has_cap_sys_ptrace()`         | euid == 0（近似检查，特殊处理）                  |

### 三、UID 转换时的能力规则 — `apply_id_change_capability_rules()`

这是 Linux `commit_creds()` 逻辑的精确模拟：

```rust
fn apply_id_change_capability_rules(&mut self, old: &Cred) {
    let old_all_root = old.euid == 0 && old.uid == 0 && old.suid == 0;
    let self_all_root = self.euid == 0 && self.uid == 0 && self.suid == 0;

    // 1. 离开"全 root" → 丢弃允许集（除非 keep_caps）
    if old_all_root && !self_all_root {
        if !self.keep_caps { self.cap_permitted = 0; }
        self.cap_effective = 0;
        self.cap_ambient = 0;
    }

    // 2. 丢失 euid=0 → 清除有效集和环境集
    if old.euid == 0 && self.euid != 0 {
        self.cap_effective = 0;
        self.cap_ambient = 0;
    }

    // 3. 重新获得 euid=0 → 恢复有效集为允许集
    if old.euid != 0 && self.euid == 0 {
        self.cap_effective = self.cap_permitted;
    }

    // 4. 强制执行不变量
    sanitize_capabilities(self);
}
```

**`sanitize_capabilities()` 确保**：
- `effective ⊆ permitted`
- `ambient ⊆ permitted ∩ inheritable`

### 四、系统调用实现 — `syscall/task/ctl.rs`

#### 1. capget(2) — 读取能力集

**流程**：
1. 读取用户空间 `__user_cap_header_struct`（version + pid）
2. 版本探测：若 `version != CAPABILITY_VERSION_3`，写入优选版本，返回 EINVAL
3. 若 data 为 NULL → 纯版本探测，返回 0
4. 从当前线程 `cred()` 读取 effective/permitted/inheritable
5. 通过 `cap_data_from_cred()` 将 u64 拆分为两个 u32 字（V3 ABI）
6. 写入用户空间

#### 2. capset(2) — 设置能力集

**严格的权限检查**（与 Linux 一致）：

| 步骤                             | 检查                            | 失败返回 |
| -------------------------------- | ------------------------------- | -------- |
| 只能修改自己                     | pid != 0 && pid != thread.tid() | EPERM    |
| 有效集不能越界                   | effective & !permitted != 0     | EPERM    |
| 不能增加允许集                   | adds_permitted != 0             | EPERM    |
| 增加到可继承集（有 CAP_SETPCAP） | 须在边界集内                    | EPERM    |
| 增加到可继承集（无 CAP_SETPCAP） | 须在当前可继承或允许集内        | EPERM    |

#### 3. prctl(2) 能力操作

| 操作              | 行为                                                         |
| ----------------- | ------------------------------------------------------------ |
| `PR_GET_KEEPCAPS` | 返回 cred.keep_caps                                          |
| `PR_SET_KEEPCAPS` | 设置 keep_caps = (arg2 != 0)                                 |
| `PR_CAPBSET_READ` | 查询能力 arg2 是否在边界集中                                 |
| `PR_CAPBSET_DROP` | 需 CAP_SETPCAP；从 cap_bounding 和 cap_ambient 中移除能力位（不可撤销） |
| `PR_CAP_AMBIENT`  | IS_SET（查询）、RAISE（提升）、LOWER（降低）、CLEAR_ALL（全部清除） |

#### 文件系统 (`syscall/fs/ctl.rs`)

### 六、与用户命名空间的集成

写入 `/proc/<pid>/uid_map` 时触发能力重算：

```rust
if orig == 0 {
    // 映射到 root → 授予全部能力
    cap_permitted = CAP_MASK;
    cap_effective = CAP_MASK;
    cap_bounding  = CAP_MASK;
} else {
    // 映射到非 root → 清除能力
    cap_permitted = 0;
    cap_effective = 0;
    cap_ambient   = 0;
}
```

`user_ns_overflow_uid()` / `user_ns_overflow_gid()`：若 `is_root` 或 `uid_mapped` 则返回 0，否则返回 65534（nobody）。

### 九、架构总结

```
                    ┌─────────────────────────────────┐
                    │          Cred (cred.rs)          │
                    │  cap_inheritable  ← exec 传递    │
                    │  cap_permitted    ← 能力上限      │
                    │  cap_effective    ← 当前有效      │
                    │  cap_bounding     ← 绝对上限      │
                    │  cap_ambient      ← 非特权保留    │
                    │  keep_caps        ← KEEPCAPS 标志 │
                    └──────────┬──────────────────────┘
                               │
          ┌────────────────────┼────────────────────┐
          │                    │                    │
    设置路径               检查路径               转换路径
    capset(2)             ~20 处检查点          setuid/setgid
    prctl(PR_CAP_*)       文件系统/网络/资源    uid_map 写入
    uid_map 写入                                   │
          │                               apply_id_change_
          ▼                               capability_rules()
    ┌──────────┐                         ──→ keep_caps 保留
    │ 权限模型  │                         ──→ euid 变化清除
    │ 只能降级  │                         ──→ sanitize() 不变量
    │ 不能提权  │
    └──────────┘
```

**核心设计原则**：
- **只能降级，不能提权** — capset 永远不能向 permitted 增加能力位
- **不变量强制执行** — effective ⊆ permitted、ambient ⊆ permitted ∩ inheritable
- **同进程线程同步** — set_cred() 更新同一进程所有线程
- **用户命名空间感知** — uid_map 映射驱动能力授予/撤销

### 四、StarryOS 内核的 runc 兼容性层

为了让真实的 `runc` 二进制在 StarryOS 上运行，内核实现了大量 runc 特定的兼容性处理：

#### 1. cgroup v2 兼容 — `cgroup/mod.rs`

- `"cgroup.kill" => Ok(())` — 接受操作但不执行资源强制
- 实现功能性的 cgroup v2 层次结构（创建节点、移动进程）
- 存储配置但不强制执行限制
- 支持 23 个 cgroup v2 接口文件
- 通过 cgroupfs 伪文件系统在 `/sys/fs/cgroup` 呈现

#### 2. procfs 兼容 — `pseudofs/proc.rs`

| 特性                 | 说明                                      |
| -------------------- | ----------------------------------------- |
| /proc 根 inode=1     | "procfs 根目录必须为 inode 1 以兼容 runc" |
| /proc/filesystems    | runc/mount 探测支持的文件系统类型         |
| /proc/self/fd/*      | PATH fd 操作，runc 用于设备节点操作       |
| /proc/self/mountinfo | runc 验证挂载状态                         |

#### 3. 系统调用兼容

| 系统调用/操作                | 文件         | 说明                                             |
| ---------------------------- | ------------ | ------------------------------------------------ |
| pivot_root(2)                | mount.rs     | "容器 init 进程可能已从 runc 克隆了挂载命名空间" |
| readlinkat + 尾部 / → EINVAL | ctl.rs       | "securejoin/runc 探测 procfs 魔术链接时使用"     |
| fchmodat2 + AT_EMPTY_PATH    | ctl.rs       | "runc 更新新创建设备节点时使用此形式"            |
| CLONE_FS                     | namespace.rs | "足以满足 runc 的锁定挂载重新映射线程"           |

#### 4. 文件系统根 inode — `pseudofs/fs.rs`
```rust
// 文件系统根目录始终报告 inode 1（runc 和其他容器运行时需要）
SimpleFs::new_with(...)
```

# 3.总结与收获

在这次项目中，我在第一阶段syscall的测例编写前对ai的使用还很浅，于是在编写测例时开始学习ai怎么写代码。这个过程并不是一帆风顺的，首先配置虚拟机环境就不是一个简单的过程，在github拉取仓库和下载必要编译链就因网络问题浪费了一些时间。而后开始使用ai编写代码。我开始使用的是claude code+deepseek api的做法，这个组合让我顺利完成了对第一阶段的任务，也让我明白了ai在代码领域的强势，未来发展趋势。而后开启了第二阶段的代码编写，docker的实现。docker是个很复杂的系统，所以我让ai帮我写了一份计划表，依照计划表来对代码进行编写。一开始是实现namespace，当我使用claude+deepseek时发现了这个组合的短处，它对代码结构理解，bug纠错等不如codex或原生claude，于是为了追求更好的代码质量，我更换为了codex。namespace我依照Debin老师建议单独做成了一个模块，然后再实现各个namespace。但因为底层vfs的原因，对mnt namespace的实现不够充分，为了进度考虑，就先暂时挂着，以后再来修复。ai在写完代码后我会认真阅读，搞清楚每个函数是干什么的，为什么要这个函数，这样才能有自己的提升。在提交pr中我也对代码进行了debug，也是很大的收获。而后陆续实现了overlayfs，seccomp，capabilities等，虽然过程有曲折，但在解决过程中有收获，也是值得的。而在合作方面，因为docker包含了很多东西，cgroup是其他人在实现，在这过程中可能有重复做的部分或依赖他人实现的部分，这需要经过交流才能解决，我于是与宋红，龙威力，刘柯，debin助教交流，他们在代码外也有对我的帮助。团队协作也是能力的一部分。遗憾最后只进行了oci和runc测试，没来得急完整实现docker，但后面有机会一定会将他实现完整。而后也更要学习，怎么驾驭如此强大的ai，使他为我所用。
