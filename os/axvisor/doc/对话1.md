当我们进行到哪一个阶段，可以尝试在qume启动linux，让linux启动进入到shell中？
星期一17:08

按当前阶段拆分，最早可以尝试 QEMU 启动 Linux 是阶段 5，但要让 Linux 稳定进入 initramfs shell，通常要到 阶段 6 才更现实。

具体判断：

阶段 3：构造 boot_params

还不能真正启动，只是把 Linux 需要的 zero page、e820、initrd 信息准备好。
阶段 4：Linux boot stub + vCPU 初始状态

这是第一次可以尝试“让 vCPU 跳进 Linux 入口”的阶段。
阶段 4 完成后，可以尝试 QEMU 启动，看能不能进入 Linux early boot。
但此时很可能卡在 early console、timer、interrupt、APIC/PIT/HPET 等位置。
阶段 5：early console + initramfs 最小闭环

这是计划里第一个“可观察启动闭环”。
如果内核 command line 内建了 console=ttyS0 rdinit=/init，并且 /init 足够简单，那么阶段 5 完成后就应该可以尝试看到 Linux early log，并进入 initramfs /init。
如果 /init 是 shell，比如 busybox sh，理论上这里就可以尝试进 shell。
阶段 6：timer、中断和基础平台设备

如果目标是“比较稳定地进入 shell 并能交互”，阶段 6 更稳。
Linux 很容易依赖 timer interrupt、LAPIC/IOAPIC/PIT/HPET/RTC 等早期平台能力；这些不完整时，可能能打印一点 log，但 shell 交互不稳定或直接卡住。
所以一句话：

阶段 4 后可以第一次试启动；阶段 5 后可以尝试进入 initramfs shell；阶段 6 后才更有把握稳定进入 shell。
