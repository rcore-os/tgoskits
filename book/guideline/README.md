# TGOSKits 指导手册

本手册面向 TGOSKits 的开发者、维护者和 PR 评审者，用于沉淀编码、设计、验证和评审中的共同准则。它不是宣传文案，也不是替代具体子系统文档的全集；它更像进入代码前的工作约束：先明确边界，再写实现，最后用可复现证据证明改动成立。

## 目标读者

| 读者 | 使用方式 |
|------|----------|
| 新功能作者 | 在设计 crate、模块、接口和测试前阅读，避免把职责、状态和边界混在一起 |
| Bug 修复作者 | 在定位根因、补回归测试、收敛错误路径时阅读，确保修复不是绕过症状 |
| PR 评审者 | 按手册检查可维护性、正确性、安全性、硬件/ABI 影响和验证证据 |
| 子系统维护者 | 用手册统一长期演进方向，减少隐性约定只存在于个人经验中 |

## 阅读入口

- [代码质量约束准则](code-quality.md)：关注 Rust 代码如何命名、拆分、组织、复用、处理错误、管理 unsafe、安排测试和自查 PR。
- [内核功能开发准则](kernel-features.md)：关注启动、平台、调度、内存、系统调用、文件系统、驱动、IRQ、DMA、虚拟化、观测和验证策略。

## 使用方式

1. 写代码前，先确认这次改动属于哪个领域边界：组件、驱动、内存、虚拟化、平台、ArceOS、StarryOS、Axvisor 或工具链。
2. 设计接口时，先查代码质量准则中的模块拆分、对象职责、错误处理、unsafe 和测试要求。
3. 实现内核能力时，按内核功能准则确认启动顺序、资源所有权、并发上下文、硬件边界、ABI 兼容和验证路径。
4. 提交 PR 前，用两个专题页的检查清单自查：入口是否清楚，职责是否单一，错误是否显式，验证是否覆盖真实风险。
5. 评审 PR 时，把本手册作为共同标准；如果上下文压缩、长时间切换任务或记不清细节，应重新阅读相关文件。

## 手册边界

| 内容 | 归属 |
|------|------|
| Rust 源码组织、函数顺序、模块拆分、对象职责、复用方式 | 代码质量约束准则 |
| 启动、调度、内存、syscall、文件系统、驱动、虚拟化、验证策略 | 内核功能开发准则 |
| 具体命令、CI 细节、板卡操作、PR 工作流 | 仓库 `AGENTS.md` 和 `.claude/skills/` 下的专题 skill |
| 用户手册、网站导航、外部发布文档 | 仓库现有 `docs/` 或后续文档系统 |

## 基本要求

- 手册中的规则是开发和评审的共同约束，不只是在 review 阶段才检查。
- 如果规则之间存在冲突，优先遵循更具体、更接近当前子系统风险的规则。
- 如果确实需要偏离规则，PR 必须说明原因、风险、替代方案和验证证据。
- 文档本身也应保持可执行、可检查、可追踪；避免只写抽象口号。
